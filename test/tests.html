Liste exhaustive des tests à mettre en place pour valider l’application en fonction des objectifs mentionnés (fonctionnalité, performance, sécurité, qualité du code, base de données).

1. Tests Fonctionnels

1.1 Authentification

  • Test de l'inscription avec données valides/invalides

  • Test de la connexion avec JWT (valide/invalide/expiré)

  • Test du rafraîchissement de token si mis en place

  • Test de la déconnexion

  • Accès interdit à une route protégée sans token

1.2 Navigation et Affichage

  • Affichage correct des pages : accueil, dashboard, séances, plans, exercices

  • Navigation entre pages (React Router)

  • Barre de recherche fonctionnelle (avec filtre)

  • Pagination (si existante)

1.3 Gestion des données utilisateurs

  • Affichage des séances/exercices/plans

  • Ajout d’un élément à une liste personnelle (mylists)

  • Suppression/modification d’un élément de liste

  • Création de listes personnelles

  • Affichage d'une liste et des items liés

1.4 API RESTful (Backend Laravel)

  • CRUD complet pour chaque entité (users, plans, workouts, exercises, swim_sets)

  • Vérification des relations : ex. ajout d’un exercice à une séance

  • Contrôle des permissions selon les rôles (roles table)

  • Vérification des réponses HTTP (200, 401, 404, 422…)

2. Tests de Performance

2.1 Frontend

  • Temps de chargement initial (First Paint, Time to Interactive)

  • Réactivité lors de l’ajout/suppression d’éléments

  • Comportement sur mobile/tablette (Responsive)

2.2 Backend

  • Temps de réponse des endpoints critiques

  • Gestion de la charge (stress test avec 50-100 utilisateurs simultanés)

  • Profilage des requêtes MySQL (requêtes lentes)

3. Tests de Sécurité

3.1 Authentification

  • JWT stocké de manière sécurisée (localStorage ou cookie httpOnly)

  • Vérification de l’expiration des tokens

  • Accès non autorisé bloqué

3.2 Vulnérabilités classiques

  • Injection SQL (via champs de recherche ou formulaire)

  • XSS (Cross-Site Scripting) dans les inputs utilisateurs

  • CSRF (si cookie utilisé pour JWT)

3.3 API

  • Endpoint protégé mal configuré (accès public à /api/users, etc.)

  • Test de modification de ressources sans permission (ex. PUT un autre utilisateur)

4. Tests de Qualité du Code

4.1 Frontend (React)

  • Composants fonctionnels bien structurés

  • Props validées (PropTypes ou TypeScript si utilisé)

  • Séparation logique UI/State/Services

  • Utilisation correcte des hooks (useEffect, useState…)

4.2 Backend (Laravel)

  • Routes bien définies et protégées (middlewares)

  • Services/Controllers bien séparés

  • Utilisation des Eloquent relationships

  • Validation côté backend

5. Tests Base de Données

  • Intégrité référentielle (via foreign keys)

  • Cohérence des relations entre entités (ex. plan_workouts, workout_exercises)

  • Données de test cohérentes et représentatives

  • Requêtes optimisées (éviter les N+1 queries)

6. Tests E2E (End to End)

  • Utiliser Cypress ou Playwright pour simuler un parcours complet :

    • Connexion

    • Recherche d’un exercice

    • Ajout à une séance

    • Ajout de la séance à un plan

    • Enregistrement dans une liste personnelle

Recommandations de Structuration

  • Frontend : utiliser Jest + React Testing Library

  • Backend : utiliser PHPUnit (Laravel intégré)

  • API : Postman/Newman pour automatisation ou Laravel HTTP Tests

  • E2E : Cypress conseillé pour simulation utilisateur complète
